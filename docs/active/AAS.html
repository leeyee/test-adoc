dm_ledger_transactions:

记录和钱有关的交易。

  • 每个production(tuition/sessionoption, merchandise) 都会有一笔不同类型的流水; 'SALE'
  • 总支付总额也会有流水，同时对于支付总额; "PAYMENT"
  • 每笔交易的交易费也会产生流水; "FEE"

external_order_id/external_order_id_INT : 对应OMS order_id
order_reference_id/order_reference_id_INT: 对应OMS order_line_id

TAB: Payment activity

Amount: 订单支付总金额
Processing Fee: 订单支付总金额FEE（用户费用+信用卡交易费）

生成临时表 #order_line_delta 数据时，delta_amount 字段使用了如下的计算

ad1.payment_amount - lag((payment_amount), 1, 0)
	OVER(PARTITION BY al1.order_line_id, ad1.payment_id, ad1.payment_type
	    ORDER BY ad1.created_dt ) AS delta_amount

但在实际查看数据过程中，发现表order_line_pmt_alloc_details.payment_amount_delta 和该数据相同。

Memebership 对应Season信息在USP中使用mock数据， ID=0, Name='Membership'

SELECT 0            AS grouping_id,
       pm.order_id,
       pm.id,
       0            AS is_waitlist,
       pm.person_id,
       NULL         AS session_id,
       pm.is_online,
       pm.order_line_id,
       NULL         AS transfer_type,
       'Membership' AS grouping_name
FROM camps_service.dbo.person_memberships pm

如果是POS，那么也不存在Season, 因此需要为Season也Mock ID 和 Name信息，如果返回为空的话，会导致AUI的 for Season Filter有异常。AccountStatementView.js#buildSeasonList

POS

Agency同步信息给POS。用户在POS上购买

  1. 如果用户购买的POS是对应Agency下的，那么Payment activity 会展示用户姓名，并且点击订单可以查看订单详情
  2. 如果用户购买的POS不是对应Agency下的，那么Payment activity 不会展用户姓名，并且点击订单查看详情的地址是错误的。

上述两种情况下，Payment activity:
+ Season 无法展示的，因为找不到。
+ Payment method 没有，应为POS生成的订单信息中 Cart_Type字段记录为空

TAB: Registration activity/Additional purchases activity 展示不出POS订单，除非购买POS的用户是提供POS的agency下的用户。
